## 1. 项目概述

打印机状态监控系统是一个用于实时监控、管理和报告企业内部打印机及打印服务器状态的综合解决方案。该系统通过多种方式（SNMP、HTTP API、截图OCR）获取打印机状态信息，并提供直观的界面展示，同时支持自动告警和定期生成报告。

### 1.1 项目目标
- 实时监控多台打印机的在线状态和墨粉水平（30秒轮询）
- 实时监控打印服务器的在线状态
- 提供统一的管理界面，方便添加、编辑和删除打印机及打印服务器配置
- 自动生成周报告并通过邮件发送
- 支持离线/墨粉告警邮件通知
- 支持多种打印机品牌和型号
- 提供许可证管理功能

### 1.2 术语定义
| 术语 | 解释 |
|------|------|
| SNMP | 简单网络管理协议，用于网络设备的管理和监控 |
| OCR | 光学字符识别，用于从图像中提取文本信息 |
| PaperCut MF | 打印管理系统，用于监控和管理打印资源 |
| HTTP API | 基于HTTP协议的应用程序接口，用于系统间通信 |

## 3. 非功能需求

### 3.1 性能需求
- 系统应能同时监控至少10台打印机
- 状态获取响应时间应在5秒内
- 报告生成时间应在30秒内
- 后端监控服务使用异步文件 I/O + 10秒缓存，避免阻塞事件循环
- 前端使用 ErrorBoundary 防止单组件崩溃导致全页面白屏

### 3.2 可靠性需求
- 系统应具备错误处理机制，当某个打印机状态获取失败时，不影响其他打印机的状态获取
- 系统应具备自动重试机制，当网络临时故障时，能自动重试获取状态
- 系统应具备数据备份机制，定期备份打印机配置和系统设置

### 3.3 安全性需求
- JWT 密钥通过环境变量配置（`.env` 文件），未配置时自动生成随机密钥并警告
- `.env` 文件已加入 `.gitignore`，不会提交到代码仓库
- 登录接口有频率限制（15分钟内最多10次尝试）
- 登录输入长度校验（用户名 ≤128，密码 ≤256），防止超长输入 DoS
- Refresh Token 轮换：每次刷新时生成新的 refresh token，旧 token 失效
- 默认管理员账号首次登录后强制修改密码（`mustChangePassword` 标志）
- 报告预览 HTML 通过 DOMPurify 消毒，防止 XSS 攻击
- 代码中不存储真实邮箱地址或凭据
- SMTP TLS 最低版本设为 TLSv1.2
- 系统部署在内网，监控页面（Status Monitoring）无需登录即可访问，其他页面需要认证

### 3.4 可用性需求
- 系统应提供友好的用户界面，操作简单直观
- 系统应提供详细的错误信息，便于故障排查
- 系统应提供在线帮助文档，指导用户使用

### 3.5 前端性能与UI优化
| 优化项 | 说明 |
|--------|------|
| 页面可见性检查 | 浏览器标签不可见时暂停前端轮询，减少不必要的网络请求 |
| Tab 导航滚动 | Tab 导航使用 scrollable 模式，避免标签过多时文字被压缩 |

## 5. 系统架构

### 5.1 架构概述
- **前端**：React + Vite构建的单页应用，负责用户界面和交互
- **后端**：Express.js构建的API服务，负责处理前端请求、获取打印机状态、生成报告等
- **数据存储**：JSON文件存储打印机配置和系统设置
- **通信**：前端通过HTTP API与后端通信，后端通过HTTP API、SNMP等与打印机通信

### 5.2 核心流程
1. **后台监控流程**（每30秒自动执行）：
   - 后端 PrinterMonitoringService 每30秒轮询所有打印机（Ping + SNMP）
   - 后端 PrintServerMonitoringService 每30秒轮询所有已启用的打印服务器
   - 监控结果缓存在内存中，供 API 即时返回
   - 状态变更时触发告警服务（AlertService）
   - 告警邮件发送失败时，EmailRetryHelper 自动指数退避重试（最多3次）

2. **前端状态加载流程**：
   - 前端登录后立即请求 `/all-printer-status` API
   - API 优先返回缓存数据（即时响应），无缓存时才实时查询
   - 前端收到数据后立即更新 UI，避免 "Unknown" 闪烁
   - 前端每30秒后台刷新状态数据

3. **告警流程**：
   - 设备离线超过配置时间 → 发送离线告警邮件（分阶段升级）
   - 设备恢复在线 → 发送恢复通知邮件（含离线时长）
   - 墨粉低于阈值 → 发送墨粉告警邮件
   - 维护模式的设备跳过告警

4. **报告生成流程**：
   - 前端请求生成报告（或后端 ScheduledReportService 定时自动触发）
   - 后端获取所有非维护模式打印机的状态
   - 后端生成HTML格式的报告
   - 后端通过SMTP发送报告邮件
   - 定时调度使用 node-cron，支持配置每周报告发送时间和收件人

5. **维护模式逻辑**：
   - 打印机：`isTest=true` → 不在仪表板显示、不计入报告、不触发告警、不参与后端监控轮询
   - 打印服务器：`enabled=false` → 暂停Ping监控、不触发告警、重置告警状态
   - 维护模式切换时，后端自动清理该设备的在线状态缓存（previousOnlineMap）

## 6. 部署与维护

### 6.1 部署方式
- **开发环境**：前端使用`npm run dev`启动开发服务器（端口5175），后端使用`node server.js`启动API服务（端口3001）
- **生产环境（Windows Server）**：使用离线部署包，通过 NSSM 注册为 Windows 系统服务（`LocalSystem` + `SERVICE_AUTO_START`）
   - 后端服务：`PrinterStatusBackend`（运行 `node server.js`，端口3001）
   - 前端服务：`PrinterStatusFrontend`（运行 `node frontend-server.js`，端口9191，静态文件服务 + API代理到3001）
   - 部署脚本：`install-service.bat`（推荐，管理员运行）、`uninstall-service.bat`、`restart-service.bat`、`stop-service.bat`
   - 调试脚本：`start-service.bat --debug`（仅会话调试，注销/重启后停止）
   - 打包命令：`deploy/pack-offline.sh` 生成离线zip包并输出验收报告

### 6.1.1 离线部署包结构
| 文件/目录 | 用途 |
|-----------|------|
| install-service.bat | 注册 NSSM Windows 服务（推荐） |
| start-service.bat | 调试会话启动（非持久化） |
| stop-service.bat | 停止调试进程/服务 |
| restart-service.bat | 重启服务 |
| uninstall-service.bat | 卸载 NSSM 服务 |
| nssm.exe | Windows 服务管理器二进制 |
| frontend-server.js | 生产前端静态服务器（端口9191）+ API反向代理 |
| backend/ | 后端代码和依赖（含node_modules） |
| dist/ | 前端构建产物 |

### 6.2 维护要点
- 定期备份打印机配置和系统设置
- 定期检查系统日志，排查潜在问题
- 及时更新系统依赖，确保安全性
- 定期检查打印机连接状态，确保监控正常
- 使用 `GET /api/health/runtime` 进行运行态巡检（进程内存、监控轮询状态、最近错误）
- 桌面支持按 `deploy/DESKTOP-SUPPORT-RUNBOOK.md` 执行标准化排障流程
- 告警发送启用去重窗口与每小时上限，降低告警噪音
- 每月执行一次备份恢复演练，记录 RTO/RPO（参考 `deploy/BACKUP-DRILL-RUNBOOK.md`）

### 6.3 故障处理
| 故障类型 | 可能原因 | 解决方案 |
|---------|---------|----------|
| 打印机状态获取失败 | 网络连接问题 | 检查网络连接，确保打印机在线 |
| | 打印机API不可用 | 尝试使用SNMP或OCR方式获取状态 |
| | SNMP配置错误 | 检查SNMP配置，确保正确设置 |

| | 网络连接问题 | 检查网络连接，确保可以访问SMTP服务器 |
| 报告生成失败 | 数据获取失败 | 检查打印机状态获取是否正常 |
| | 模板错误 | 检查报告模板是否正确 |
